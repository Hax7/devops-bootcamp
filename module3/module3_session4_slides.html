üîó Making Smart API Calls with requests

You will often call external HTTP APIs in automation. The requests library makes it easy ‚Äì let's ensure we do it reliably.

Think üßê: What could go wrong when calling an external API from a script?
Possible issues: network timeouts, slow responses, non-200 status codes, wrong data format, authentication needed, etc. Get students to consider failure modes (this motivates why we need timeouts, error checks, etc. in code).

Example: Basic GET Request (JSON API)

import requests # GET a sample JSON resource response = requests.get("https://jsonplaceholder.typicode.com/todos/1") # Check for success if response.status_code == 200: data = response.json() # parse JSON to Python dict print(f"Todo title: {data['title']}") else: print(f"Request failed with code: {response.status_code}")
Here we use requests.get to fetch a test API (a placeholder todo item). We then check the HTTP status code (200 means OK) and use .json() to parse the JSON response into a Python dictionary:contentReference[oaicite:5]{index=5}. Note: By default, this call has no timeout ‚Äì if the server hangs, our script would hang (we'll address that next).

Making API Calls Resilient

  • Always set a timeout to avoid hanging indefinitely:contentReference[oaicite:6]{index=6} (e.g. requests.get(url, timeout=5)).
  • Use response.raise_for_status() to catch HTTP errors automatically:contentReference[oaicite:7]{index=7}.
  • Leverage response.json() to parse JSON responses into Python data:contentReference[oaicite:8]{index=8}.
  • Add necessary headers (auth tokens, content-type, etc.) as required by the API.
  • Implement retries/backoff for robustness (retry failed requests a few times rather than failing outright).
**Timeouts:** Emphasize that requests has **no default timeout**, so always specify one:contentReference[oaicite:9]{index=9}. **Errors:** raise_for_status will throw an exception for 4xx/5xx codes so you can handle it (or let your script log/exit):contentReference[oaicite:10]{index=10}. **Headers:** Example - include auth tokens (Bearer token) or set Content-Type: application/json for POST. **Retries:** Note that by default requests won't retry failed calls. You can implement a simple retry loop or use advanced techniques (like urllib3 Retry with a Session):contentReference[oaicite:11]{index=11}:contentReference[oaicite:12]{index=12}. It's also good to log API calls and responses when automating, for audit/troubleshooting.

üì° Remote Automation with fabric

The fabric library lets us run shell commands on remote servers via SSH, using Python (no manual SSH needed!).

Question: What kinds of tasks would you automate on remote servers? (Think of repetitive admin tasks.)
Discuss ideas: e.g., updating software on multiple servers, restarting a service across dozens of VMs, editing config files remotely, collecting logs, etc. Fabric (built on Paramiko) gives us a Pythonic way to do these over SSH:contentReference[oaicite:14]{index=14}. It‚Äôs like writing a script that runs commands on another machine.

Example: Updating a Web Server via SSH

from fabric import Connection # Connect to remote server (using SSH key authentication) conn = Connection(host="ubuntu@", connect_kwargs={"key_filename": "mykey.pem"}) # Run commands on the remote server conn.run("echo 'Hello from Fabric!' > index.html") conn.run("sudo mv index.html /var/www/html/index.nginx-debian.html") conn.run("sudo systemctl reload nginx") print("Update complete.")
We connect to an Ubuntu server (e.g., an EC2 instance) using an SSH key:contentReference[oaicite:15]{index=15}. Then we: ‚Ä¢ Create a file with some text (simulating a new webpage) ‚Ä¢ Move it into Nginx's web directory with sudo (requires root privileges) ‚Ä¢ Reload Nginx to apply the change After running this, the server's homepage is updated without us logging in manually. Highlight that any shell command can be executed via conn.run. Fabric captures the command output as well (which we could print or log).

üëÄ File Watchers + Auto-Backup to S3

We can automate reactions to file changes using watchdog (to monitor filesystem events) and boto3 (to interact with AWS S3).

Discussion: Why monitor files? (Imagine auto-backups for logs, syncing config changes, or triggering CI when new files appear.)
Explain common use cases: e.g., automatically uploading new log files for backup, watching a folder for config changes to restart a service, or triggering a build when a file is dropped in a directory. In DevOps, this pattern reduces manual checks and ensures immediate responses to changes.

Example: Auto Upload to S3 on New File

import boto3, time, os from watchdog.observers import Observer from watchdog.events import FileSystemEventHandler s3 = boto3.client('s3') # AWS credentials should be configured (env/profile) BUCKET_NAME = "my-demo-bucket" WATCH_FOLDER = "/path/to/watch" class UploadOnChange(FileSystemEventHandler): def on_created(self, event): if not event.is_directory: filename = os.path.basename(event.src_path) s3.upload_file(event.src_path, BUCKET_NAME, filename) print(f"Uploaded {filename} to S3!") observer = Observer() observer.schedule(UploadOnChange(), WATCH_FOLDER, recursive=False) observer.start() print(f"Watching {WATCH_FOLDER} ...") try: while True: time.sleep(1) except KeyboardInterrupt: observer.stop() observer.join()
This script watches a folder and uploads any new file to an S3 bucket:contentReference[oaicite:16]{index=16}. Walk through it: we subclass FileSystemEventHandler and override on_created. When a new file is detected, we use boto3 to upload it to BUCKET_NAME. The observer monitors the directory in a background thread. Note: Ensure students have AWS creds set (perhaps from our earlier AWS session). In real scenarios, you'd also handle file modifications or deletions, and include error handling (e.g., wrap upload_file in try/except). To stop the script, they press Ctrl+C (which triggers the KeyboardInterrupt and stops the observer).
